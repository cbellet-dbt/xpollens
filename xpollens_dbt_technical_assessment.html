📋 Synthèse Exécutive

Contexte du POC

Xpollens, acteur du secteur des paiements, souhaite évaluer dbt Platform pour moderniser sa plateforme data et autonomiser ses data analysts dans la création de tables matérialisées. L'équipe data de 6 personnes (2 data engineers, 2 data analysts) utilise actuellement une stack Microsoft (Synapse Analytics serverless, Power BI, Azure DevOps) avec une architecture en médaillon (Bronze, Silver, Gold).

💡 Opportunité

Autonomiser les data analysts pour créer des tables matérialisées sans intervention des data engineers, tout en maintenant un processus de validation (code review) et une gouvernance des données.

🎯 Solution

dbt Platform avec l'IDE intégrée et Canvas (interface low-code) permettant aux analysts de créer des modèles en SQL avec validation automatique via CI/CD et Azure DevOps.

💰 Impact Business

Réduction du temps de livraison des nouveaux modèles, amélioration de la qualité des données grâce aux tests automatisés, et documentation centralisée du Data Lake.

⚠️ État Actuel & Points de Friction

1️⃣ Dépendance Data Engineers

Problème: Les data analysts ne peuvent pas créer de tables matérialisées sans l'intervention des data engineers
Impact: Files d'attente pour les demandes, data engineers surchargés de demandes répétitives
Cause: Transformations Power BI non matérialisées, absence d'interface accessible pour les analysts

2️⃣ Performance des Requêtes Power BI

Problème: Modèle sémantique unique alimentant 90% des dashboards, temps de requête parfois longs
Impact: Expérience utilisateur dégradée, impossibilité de filtrer efficacement pour les dashboards client-spécifiques
Cause: Données non matérialisées en amont, tout calculé à la volée dans Power BI

3️⃣ Gestion des Tests de Qualité

Challenge: Tests de qualité existants sur dbt Core (~40 tables testées) mais dashboard de résultats custom
Limitation: Pas de visualisation intégrée, rapport Power BI qui parse les fichiers de résultats
Maintenance: Fichiers YAML complexes à maintenir pour les non-développeurs

4️⃣ Documentation du Data Lake

Problème: Documentation insuffisante du Data Lake
Impact: Difficultés d'onboarding, dépendance aux personnes clés
Approche actuelle: CasterDoc utilisé mais pas exhaustif

5️⃣ Règles Métier Complexes

Challenge: Règles de gestion variant selon les clients/partenaires (ex: opérations SEPA vs carte)
Exemple: Tables séparées pour opérations SEPA et carte nécessitant des unions fréquentes
Question: Faut-il unifier dans Gold ou dans le Semantic Layer?

🔧 Solution dbt Platform

Qu'est-ce que dbt?

dbt (data build tool) permet aux analytics engineers d'écrire, tester, documenter et versionner du code de transformation de données en SQL. Plus d'informations sur getdbt.com

Workflow de Validation (PR Review)

Processus de Gouvernance Obligatoire:

Tout code (IDE ou Canvas) passe par le même workflow de validation:
  1. Analyst crée/modifie un modèle SQL
  2. Clic sur "Commit" → Pull Request automatique sur Azure DevOps
  3. Data Engineer fait la code review
  4. CI/CD exécute les tests automatiques
  5. Merge → Déploiement en production

Note: Ce processus est natif à dbt Platform et ne peut pas être contourné.

Comparaison dbt Core vs dbt Platform

⚠️ Point Important: Évolution des Produits

dbt Platform est un produit en évolution constante avec des nouvelles fonctionnalités régulières. dbt Core reste un projet open-source maintenu, mais ne bénéficiera pas des innovations majeures futures. Cette distinction est importante pour la stratégie data à moyen/long terme.

Aspect dbt Core (Open Source) dbt Platform (Cloud)

Interface CLI uniquement, nécessite VS Code ou éditeur local IDE Web intégrée + CLI + VS Code Extension + Canvas (low-code)

Accessibilité Profils techniques uniquement (data engineers) Data engineers + Data analysts + Métiers (via Canvas)
Création de modèles Écriture SQL manuelle SQL manuel + Canvas drag & drop + génération assistée Copilot

Framework de tests ✅ Fichiers YAML (tests génériques + custom) ✅ Fichiers YAML + génération automatique via Copilot

Alerting À configurer manuellement (scripts, webhooks) Alerting natif (email, Slack, Teams en preview)
Health Check Power BI ❌ Non disponible ✅ Vérification de cohérence avec les rapports Power BI
📖 Documentation & Lineage
Documentation Génération statique (dbt docs generate), maintenance manuelle Documentation dynamique, auto-générée par Copilot, toujours à jour

💡 Cas d'usage dbt Core

  • Équipe 100% technique (data engineers expérimentés)
  • Infrastructure CI/CD déjà mature et maintenue
  • Budget très contraint, pas de priorité productivité
  • Pas de besoin d'autonomie des data analysts

✅ Cas d'usage dbt Platform

  • Volonté d'autonomiser les data analysts
  • Besoin de gouvernance et traçabilité
  • Focus sur la productivité et le time-to-value
  • Stratégie data à moyen/long terme
  • Besoin de documentation centralisée et à jour

🎯 Recommandation pour Xpollens

Compte tenu des objectifs identifiés (autonomie des data analysts, amélioration de la qualité, documentation du Data Lake), dbt Platform répond mieux aux besoins que dbt Core. Les fonctionnalités Enterprise (Canvas, Copilot, Semantic Layer) sont particulièrement pertinentes pour permettre à Emma et aux futurs analysts de créer des modèles sans expertise SQL avancée.

Points d'Attention & Bloquants

✅ Résolu (12/02): Matérialisation Tables Delta

Solution: Passage sur pool dédié Synapse (au lieu de serverless).
Résultat: Création de tables et modèles fonctionnels. Tests OK. Droits configurés.
⚠️ Point d'attention: Migration dbt Core → Fusion

Contexte (12/02): Olivier a souligné que dbt Labs investit principalement dans Fusion (rachat SDF ~200M$). dbt Core reste maintenu mais avec moins d'évolutions futures.
Recommandation: Intégrer la migration Core → Fusion dans la roadmap, même sans dbt Platform.
Avantage Platform: Analyse des logs et suggestions de migration automatiques (non disponible en self-hosted).
⚠️ Clarifications Low-Code (Canvas)

Constat (02/02): L'équipe pensait que les tests pouvaient être gérés via l'interface graphique Canvas.
Réalité: Canvas = création de modèles SQL (low-code). Les tests restent en fichiers YAML via IDE/VS Code.
Aide disponible: Copilot peut générer automatiquement les tests génériques et mettre à jour les fichiers YAML.
⚠️ Limitation Connecteur Synapse (19/02)

Constat: Le connecteur Synapse ne supporte pas la stratégie d'incrémentation en merge → utilisation de delete-insert.
Note: Le connecteur n'apparaît pas dans la documentation des types de stratégies supportées.
Impact: À évaluer selon les cas d'usage (tables volumineuses avec mises à jour partielles).
⚠️ Lecture Delta par Pool Dédié (19/02)

Constat: Problème de lecture des tables Delta par le pool SQL dédié.
Statut: Sujet Microsoft en cours de résolution.
⚠️ À Valider: Choix du Tier (Team vs Enterprise)

Note (12/02): Le POC est sur la version Enterprise avec Canvas, Copilot, etc.
